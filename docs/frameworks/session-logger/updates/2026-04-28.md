# Session Logger Framework Update 2026-04-28

## Summary

Added forced automatic safety capture for accepted implementation plans, then refined it to use append-only `auto_recovery.md` instead of overwriting hot summary memory.

## Why

Long, compacted, or multi-plan coding sessions were losing important BEFORE -> NOW details. The user needed the agent procedure itself to preserve decisions, assumptions, intended scope, actual changes, verification, and remaining work without relying on manual checkpoints or destructive hot-summary overwrites.

## What Changed

- Added automatic safety capture as a narrow exception to the explicit Session Logger invocation gate.
- Required a BEFORE entry before the first mutating step of an accepted implementation plan.
- Required an AFTER entry after execution and verification, before final response, `/compact`, or `/fork`.
- Required the rule for each distinct execution plan in long or multi-plan work.
- Added narrow automatic non-plan safety entries for context pressure, before `/compact`, before `/fork`, and major decisions/milestones.
- Added `docs/workflow/auto_recovery.md` as an append-only active-session recovery buffer.
- Changed automatic capture so it appends to `auto_recovery.md` instead of overwriting `last_session_summary.md`.
- Kept manual `mid` as the curated hot-summary writer.
- Made session `end` responsible for merging useful auto recovery entries into normal session memory and clearing `auto_recovery.md` only after successful incorporation.
- Defined no-primary behavior as blocking execution until the user activates or starts a primary project.
- Added registry-level access metadata for external read-only reference repositories.

## Impact On Project Structure

Projects using Session Logger should add:

```text
docs/workflow/auto_recovery.md
```

New scaffolds include this file. Existing projects should adopt it through manual framework audit or an explicit governed workflow update.

## Impact On Governed Files

Governed files affected by this update:

- `.codex/session-logger.toml` for adopted framework date tracking during manual audit
- `docs/workflow/auto_recovery.md`
- `docs/workflow/checkpoint.md`
- `docs/workflow/session_start.md`
- `docs/workflow/session_end.md`
- `docs/workflow/last_session_summary.md`

The active skill and root workspace instructions should also document automatic safety capture so future agents enforce it.

## Impact On Governed Data

`auto_recovery.md` may now receive automatic append-only entries when a primary project is active. Entries should include timestamp, source/trigger, plan/checkpoint ID when applicable, concise summary, important decisions, risks/blockers, relevant paths/files, and status.

BEFORE entries should capture prior state, intended plan, decisions/assumptions, expected scope, and risks. AFTER entries should use the same plan/checkpoint ID and capture original intent summary, actual changes, decisions made during execution, verification, remaining work, and the BEFORE -> NOW delta.

Manual `mid` may read relevant automatic entries and blend durable facts into `last_session_summary.md`. Session `end` should merge useful entries into latest session memory, dedupe them, surface unmatched BEFORE entries as unclosed plan markers, and clear `auto_recovery.md` only after successful incorporation.

This update does not change warm/cold/freezing permission requirements, allowed read-only project behavior, or archive access rules.

External read-only registry entries are stricter than normal allowed read-only activation: they can never become primary and must never receive Session Logger writes, even if old scaffold files already exist in their project tree.

## Required Audit Checks

### Project audit checks (governed surfaces in the primary active project)
- Confirm project-local `docs/workflow/auto_recovery.md` exists or is proposed as an upgrade action.
- Confirm project-local `docs/workflow/checkpoint.md` describes manual `mid` as hot-summary blending and automatic capture as append-only.
- Confirm `docs/workflow/session_start.md` surfaces unresolved automatic entries without treating them as warm/cold memory.
- Confirm `docs/workflow/session_end.md` merges and clears `auto_recovery.md` only after successful incorporation.
- Confirm no project-local text implies automatic safety capture can write to allowed read-only projects.
- Confirm external read-only projects are blocked from primary activation and all Session Logger writes.
- Record project compliance status (`up_to_date`, `review_needed`, `blocked`) and propose specific upgrade actions when compliance is incomplete.

### Shared framework self-checks (root/shared infrastructure, not project compliance)
- Confirm the active skill `.agents/skills/session-logger/SKILL.md` documents automatic safety capture and its invocation-gate exception.
- Confirm root `AGENTS.md` requires append-only automatic safety entries for accepted implementation plan execution.
- Confirm manual framework audit remains manual-only and manifest-driven.
- Confirm logger-only session-end commits require explicit user choice.
- Confirm `.agents/projects-index.json` access metadata marks external reference repositories with `access_mode: "external_read_only"` and `write_policy: "forbidden"` when applicable.

Shared self-checks must not be reported as project compliance defects.
