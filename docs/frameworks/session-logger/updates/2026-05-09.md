# Session Logger Framework Update 2026-05-09

## Summary

Closed the startup recovery gap by making `auto_recovery.md` a bounded hot-adjacent inspection surface during activation/start.

## Why

Automatic safety entries can contain work that was captured but never merged into `last_session_summary.md` by manual `mid` or session `end`. Startup procedures that read only `project_identity.md` and `last_session_summary.md` can miss unresolved automatic entries and resume from an incomplete state.

## What Changed

- Required activation/start to inspect `docs/workflow/auto_recovery.md` after `project_identity.md` and `last_session_summary.md` when the file exists.
- Limited startup inspection to unresolved automatic safety entries, unmatched BEFORE entries, `open` / `unclosed` statuses, or non-template entries that affect restart accuracy.
- Defined this inspection as hot-adjacent recovery, not warm/cold/freezing access.
- Required unresolved auto-recovery state to be surfaced before stale-hot warm recovery is requested.
- Kept resolved entries subordinate to curated hot memory unless they affect restart accuracy.

## Impact On Project Structure

No new files are required beyond the existing `docs/workflow/auto_recovery.md` introduced by the 2026-04-28 update.

Projects missing `docs/workflow/auto_recovery.md` should still receive a specific audit upgrade action; startup must not silently create it.

## Impact On Governed Files

Governed files affected by this update:

- `.codex/session-logger.toml` for adopted framework date tracking during manual audit
- `docs/workflow/session_start.md`
- `docs/workflow/auto_recovery.md`

The active skill, help text, root workspace instructions, and shared framework README should also document the startup inspection rule so future agents enforce it.

## Impact On Governed Data

This update does not change the automatic safety entry schema.

Startup may read only enough of `auto_recovery.md` to identify unresolved or restart-relevant automatic entries. It must not treat `auto_recovery.md` as archive history, warm memory, cold memory, freezing memory, or a replacement for `last_session_summary.md`.

Manual `mid` and session `end` remain responsible for merging useful automatic entries into normal session memory and deduping against existing facts.

## Required Audit Checks

### Project audit checks (governed surfaces in the primary active project)
- Confirm project-local `docs/workflow/session_start.md` inspects `auto_recovery.md` during startup for unresolved or restart-relevant automatic entries.
- Confirm project-local startup wording treats `auto_recovery.md` as hot-adjacent recovery, not warm/cold/freezing memory.
- Confirm project-local startup wording surfaces unresolved auto-recovery state before asking for stale-hot warm recovery.
- Confirm project-local startup wording does not imply resolved automatic entries override curated hot memory.
- Confirm project-local `docs/workflow/auto_recovery.md` exists or propose a specific upgrade action.
- Record project compliance status (`up_to_date`, `review_needed`, `blocked`) and propose specific upgrade actions when compliance is incomplete.

### Shared framework self-checks (root/shared infrastructure, not project compliance)
- Confirm `AGENTS.md` requires bounded `auto_recovery.md` startup inspection.
- Confirm `.agents/skills/session-logger/SKILL.md` requires bounded `auto_recovery.md` startup inspection.
- Confirm `.agents/skills/session-logger/HELP.md` explains the startup inspection behavior.
- Confirm `docs/frameworks/session-logger/README.md` documents the startup inspection behavior.
- Confirm the shared `docs/frameworks/session-logger/templates/workflow/session_start.md` remains aligned with this behavior.

Shared self-checks must not be reported as project compliance defects.
